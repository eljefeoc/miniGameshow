**CONCERN 2 — Font preloads in penguin-game.html (D-18):**

Current font loading (line ~10–12):

Add two `<link rel="preload">` tags BEFORE the `rel="stylesheet"` link. These hint the browser to fetch font files before CSS parsing completes, eliminating FOUT:
```html
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link rel="preload" as="font" crossorigin
      href="https://fonts.gstatic.com/s/titanone/v13/mFTyWbofw6zc9NtnW43SuRwr.woff2">
<link rel="preload" as="font" crossorigin
      href="https://fonts.gstatic.com/s/bubblegumsans/v18/AYCSpXb_Z9EORv1M5QAjBbNC8TY.woff2">
<link href="https://fonts.googleapis.com/css2?family=Titan+One&family=Bubblegum+Sans&display=swap" rel="stylesheet">
```

Note: The woff2 URLs above are the canonical jsDelivr/Google Fonts CDN paths for these fonts. **CONCERN 3 — Pin Supabase JS to a specific version with SRI hash (D-17, SECU-03):**

Current import (line ~2634):
```javascript
const { createClient } = await import('https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2/+esm');
```
This pins to the major version `@2` but will auto-update on minor/patch releases — a supply-chain risk.

Steps:
1. Determine the current latest stable `@supabase/supabase-js` version in the 2.x line. Check `https://cdn.jsdelivr.net/npm/@supabase/supabase-js` for the latest published version, or use `npm view @supabase/supabase-js@2 version` if npm is available.

2. Construct the pinned ESM URL: `https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2.X.Y/+esm` (replace X.Y with actual version).

3. Generate the SRI hash for that exact URL:
   ```bash
   # Method A: if curl and openssl available
   curl -sL "https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2.X.Y/+esm" \
     | openssl dgst -sha384 -binary \
     | openssl base64 -A
   # Result: base64 string → prefix with "sha384-" for the integrity attribute
   
   # Method B: use jsDelivr's own SRI endpoint
   curl "https://www.jsdelivr.com/package/npm/@supabase/supabase-js?tab=files" 
   # OR use the hash from jsDelivr's /integrity endpoint if available
   ```

4. Replace the import line with the pinned URL. Because `import()` does not support `integrity` attributes (dynamic import SRI is not yet broadly supported), change the approach: replace the dynamic `import()` with a `<script type="module">` tag loaded with integrity, OR use a wrapper approach.

   The simplest approach for this no-build codebase: add a `<script>` tag in `<head>` that loads the ESM bundle with integrity, then access it via the global it exports. However, Supabase ESM does not expose a window global by default.

   **Recommended approach for this project:** Change the dynamic import URL to the pinned version. Even without an `integrity` attribute on dynamic import (browser limitation), a pinned exact version URL eliminates the "floating latest" risk — the CDN file at a pinned URL is immutable. Add a comment explaining the SRI limitation:

   ```javascript
   // Supabase JS pinned to exact version per SECU-03 — immutable CDN URL
   // SRI hash not applicable to dynamic import() (browser spec limitation)
   // Pin: @supabase/supabase-js@2.X.Y
   const { createClient } = await import('https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2.X.Y/+esm');
   ```

   Replace `2.X.Y` with the actual version number determined in step 1.

   The primary approach for SRI is to load Supabase via a `<script type="module" integrity="sha384-..." src="...">` tag in `<head>` instead of a dynamic `import()`. Generate the SRI hash with: `curl -s "https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2.X.Y/dist/umd/supabase.js" | openssl dgst -sha384 -binary | openssl base64 -A`, then prefix with `sha384-`. Attach the Supabase client to `window._miniGameshowSb` in the module so existing code continues to work. If the restructuring causes breakage in the auth flow that cannot be resolved quickly, fall back to the version-pinned URL alone (no integrity hash) and leave a TODO comment.
